## The blocker

**Pixel-clock timing margin is too tight on the new bitstream containing `fio.sv`.**

- Yesterday's seed-5 bitstream (`build/e6502.bit`, md5 `cc554b6743f3cbee0529eefb562f6a92`) closes at 27.41 MHz vs 25 MHz target — looks fine on paper.
- BUT same `.bit` flashed twice produces different visual output: first time clean, second time the ~50px black stripe on the left returns.
- BUT integration sweep wedges mid-run: `cursor_x:42, cursor_y:42` (out-of-bounds) appears, all subsequent commands time out.
- BUT single tests run via `--test <name>` pass cleanly.

We've ruled out:
- The fio.sv RTL itself (nothing references SD pins from FPGA — `ulx3s.lpf` doesn't constrain them)
- The cpu_din priority mux being too deep (already 2-level grouped per commit `ddfba5a`)
- The error-detection change in sim6502 (problem persists with it stashed)
- A specific COPPER test triggering the wedge (it now reproduces from boot)
- Routing seed luck (seeds 1-5 tested; even the best — seed 5 at 27.41 MHz — is unstable)

Root cause hypothesis: the BRAM tile decode path for `ram_a_dout` is structurally ~12-15 ns deep (5.83 ns clk-to-q + 32-tile decode tree), and synth treats it as a 25 MHz path even though `cpu_ce` toggles every clock so the CPU effectively consumes `cpu_din` at 12.5 MHz. The correct relaxation is multicycle.

## What to do first (low-risk, high-info)

**Try an SDC multicycle constraint.** nextpnr-ecp5 supports `--sdc <file>`. The path from BRAM regs → cpu_din mux → CPU register only matters when cpu_ce is high (every other clock), so a 2-clock multicycle should be safe.

```bash
cd /Users/barry/Git/e6502/e6502.FPGA/fpga
cat > cpu_din.sdc <<'EOF'
# CPU consumes cpu_din only when cpu_ce is high (every other clk_pixel).
# Allow 2 clocks for the BRAM-decode → cpu_din-mux → CPU-register path.
create_clock -name clk_pixel -period 40.0 [get_ports clk25_mhz]
set_multicycle_path -setup 2 -from [get_clocks clk_pixel] -to [get_pins cpu_inst/*]
set_multicycle_path -hold 1 -from [get_clocks clk_pixel] -to [get_pins cpu_inst/*]
EOF
# add --sdc cpu_din.sdc to the nextpnr command in Makefile, then:
rm -f build/e6502.{json,config,bit}
make bitstream 2>&1 | tee /tmp/synth.log
grep "MHz" /tmp/synth.log | tail
openFPGALoader -b ulx3s build/e6502.bit
sleep 2
python3 /Users/barry/Library/Arduino15/packages/esp32/hardware/esp32/3.3.8/tools/espota.py \
    -i 192.168.1.65 -p 3232 -f /tmp/novahost-build/novahost.ino.bin
```

Expected: post-opt clk_pixel report should show much higher fmax (the path now budgeted as 80ns instead of 40ns). Visual artifact gone, sweep runs cleanly.

## Plan B — register cpu_din with cpu_ce-aware re-timing

```verilog
// In top.sv, replace the always_comb cpu_din mux with:
always_ff @(posedge clk) begin
    if (cpu_ce_pre)  // fires one clk before cpu_ce
        cpu_din <= cpu_din_combinational;
end
```

Where `cpu_din_combinational` is the existing 3-way priority mux. This puts `cpu_din` on a register with the upstream path ending at the register's D input. New register fires only on the cycle BEFORE cpu_ce so that data is stable when CPU advances.

Risk: changes CPU read timing. Need to verify with full sim suite (FPGA verilator tests) before flashing.
