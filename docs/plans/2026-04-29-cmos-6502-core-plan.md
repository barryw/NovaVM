# CMOS 65C02 Core Bring-Up Plan

Started: 2026-04-29

Purpose: make the FPGA CPU core match NovaVM's existing 65C02-class software contract, then use the stable CMOS instruction set to shrink and simplify ROM code.

## Goals

- Make FPGA and Avalonia CPU behavior match for documented NovaVM programs.
- Pass the existing NMOS Klaus Dormann functional and interrupt suites.
- Pass the existing 65C02 extended opcode suite.
- Keep timing deterministic on ULX3S after each CPU-core milestone.
- Only optimize Nova ROMs after the hardware core and tests are stable.

## Non-Goals

- Do not invent Nova-specific CPU instructions in this phase.
- Do not replace Arlet's core unless incremental extension becomes clearly worse than replacement.
- Do not rely on seed hunting to make a larger decode path route.
- Do not rewrite ROMs while the hardware CPU contract is still moving.

## Current State

- FPGA uses Arlet Ottens' compact 6502 core in `e6502.FPGA/rtl/arlet_6502/cpu.v`, now extended with the tested 65C02 opcode set.
- The core passes the Klaus Dormann functional harness, the CMOS-adjusted interrupt harness, and the 65C02 extended opcode harness.
- The Verilator CPU regression targets are wired into `e6502.FPGA/test/Makefile`.
- Avalonia already models a CMOS CPU mode and has an opcode list that includes 65C02 instructions.
- ROM build steps already use `ca65 --cpu 65c02`, so the toolchain is ready before hardware is.

## Baseline Failure

Command:

```sh
make -C e6502.FPGA/test test_cpu_dormann_65c02
```

Initial result before implementation:

- Times out after `500000000` cycles.
- Final `PC=$0424`, executing the self-loop for a failed `BNE`.
- The failing early sequence tests `PHX`: `LDX #$55`, `PHX`, `LDX #$AA`, `PHX`, `CPX $01FE`, `BNE *`.
- This confirms the current NMOS core is not executing the CMOS stack opcodes and gives a clean first implementation slice.

## Acceptance Tests

Required before using CMOS opcodes in ROM:

- `make -C e6502.FPGA/test test_cpu_dormann_6502`
- `make -C e6502.FPGA/test test_cpu_dormann_interrupt`
- `make -C e6502.FPGA/test test_cpu_dormann_65c02`
- Targeted Verilator tests for each new instruction family.
- Full FPGA synthesis with timing check margin after CPU decode changes.

The 65C02 Dormann target now passes and should remain in the normal CPU regression set.

## Implementation Phases

### Phase 1: Test Harness And Diagnostics

- Run the current 65C02 Dormann target and capture the first failure location.
- Add trace controls or failure decoding if the existing timeout output is not enough.
- Add small focused tests for individual 65C02 opcode families so failures are local and fast.
- Keep NMOS Dormann tests passing throughout.

### Phase 2: Low-Risk CMOS Instructions

Implement instructions that mostly reuse existing datapath/control:

- `BRA`
- `STZ` zeropage, zeropage,X, absolute, absolute,X
- `PHX`, `PLX`, `PHY`, `PLY`
- `INC A`, `DEC A`
- `BIT #imm`, `BIT zp,X`, `BIT abs,X`
- CMOS-defined NOP lengths and addressing forms needed by tests.

These should cost little or no new datapath width. Main risk is decode fanout.

### Phase 3: New Addressing Modes

Implement addressing behavior that needs new or adjusted microstates:

- `(zp)` indirect addressing for `ADC`, `AND`, `CMP`, `EOR`, `LDA`, `ORA`, `SBC`, `STA`.
- `JMP (abs,X)`.
- Correct CMOS `JMP (abs)` behavior without the NMOS page-wrap bug.

These changes affect PC/address sequencing and need focused tests before Dormann retry.

### Phase 4: Bit And Read-Modify-Write Instructions

Implement bit-oriented instructions:

- `TRB` zeropage and absolute.
- `TSB` zeropage and absolute.
- `RMB0` through `RMB7`.
- `SMB0` through `SMB7`.
- `BBR0` through `BBR7`.
- `BBS0` through `BBS7`.

These are the highest semantic lift because they combine memory read, bit test/mutate, writeback, and sometimes branch control.

### Phase 5: Interrupt And Stop/Wait Semantics

Implement or explicitly define:

- IRQ/NMI/BRK/RESET clear decimal mode on CMOS.
- Decimal-mode ADC/SBC flag behavior matches the software model.
- `WAI` behavior.
- `STP` behavior.

If NovaVM does not need true low-power stop, `STP` can be defined as a controlled CPU halt that debug/reset can recover from.

### Phase 6: Integration And Timing

- Run all CPU regression tests.
- Run relevant top-level FPGA tests that execute ROM or MMIO code.
- Synthesize after each decode expansion milestone.
- If timing regresses, split decode into registered family flags or simplify hot-path muxing before adding more features.

## ROM Upgrade Pass

Start only after the CMOS core is accepted.

Candidate byte-saving upgrades:

- Replace branch-around patterns with `BRA`.
- Replace `LDA #0` / `STA addr` with `STZ addr`.
- Replace `TXA` / `PHA` and `PLA` / `TAX` with `PHX` / `PLX`.
- Replace `TYA` / `PHA` and `PLA` / `TAY` with `PHY` / `PLY`.
- Use `(zp)` addressing where it removes setup instructions.
- Use `TRB` / `TSB`, `RMB` / `SMB`, and `BBR` / `BBS` for bitfield-heavy ROM paths.

Process:

- Establish ROM size and test baseline first.
- Optimize one routine family at a time.
- Keep changes measurable: bytes saved, cycles changed, tests run.
- Prefer clarity when the byte saving is tiny and the routine is not hot or space-sensitive.

## Risks

- CPU decode is part of a known timing-sensitive system. Adding opcode families may move the critical path.
- Dormann passing does not prove exact cycle behavior for every Nova workload, so small directed tests still matter.
- Avalonia and FPGA may disagree on underdocumented behavior unless both are tied to the same written CPU contract.
- ROMs already assemble as 65C02, so accidental future use of CMOS opcodes before hardware support could brick boot paths unless tests catch it.

## Open Decisions

- Decide whether NovaVM targets WDC 65C02 behavior exactly or a documented Rockwell/WDC practical superset.
- Decide exact `WAI` and `STP` behavior for NovaVM.
- Decide whether to keep an NMOS compatibility mode. Default recommendation: no, make NovaVM CMOS-only once the core is stable.
