## Known problems, ordered by blast radius

### 1. DMA suite 5/5 FAIL on hardware (regression)

`DMACOPY 0,$2000,0,$3000,8` returns 0xAA (untouched destination),
expected the copied source bytes. Tested:

```
dmacopy-cpu-to-cpu - DMACOPY within CPU RAM : FAILED
  Expected 170 == 10 in assertion 'byte 0 copied'
```

Sim passes 79/79 blitter tests. This case has height=1 so the new
`src_row_off/dst_row_off` accumulator path should be inert — initial
0, never incremented. But the hardware disagrees.

**Debug starting points:**
- Bitstream was synthesized from `e45706e` tree, diag 202/208
- nextpnr warned on one unrelated clock (check pnr log); pixel clock is
  25.83 MHz so the blitter path itself closed
- Could be: (a) a sim/synth behavioral divergence on the accumulator
  init, (b) blitter status register reporting stale OK from prior blit
  (CPU-side polling issue), (c) width/height=8/1 edge case I missed
- First move: run DMACOPY under Verilator `test_rom_load`-style harness
  with the same BASIC sequence and compare signals

### 2. SID 6581 filter broken on hardware (expected, known)

`f6581_curve` moved to SDRAM port B in `sid_curve_reader.sv`. ESP32
NovaHost doesn't yet stream the curve to SDRAM at boot. Until it does,
SID 6581 filter reads zero/garbage. 8580 linear mode still works.

Also failing: `sid2-freq-write` — probably independent, check.

**Next:** Extend NovaHost with a `poke_sdram` command and teach its
boot-time ROM loader to also stream `e6502.FPGA/rom/f6581_curve.hex`
(4096 entries × 2 bytes, big-endian? little-endian? check
`sid_curve_reader.sv:46-48`) to CURVE_BASE = `25'h0_8_00_00`.
FPGA side is already waiting.

### 3. `fileio` suite 5/5 FAIL on hardware (likely NovaHost, not FPGA)

`GSAVE`/`GLOAD`/`DEL` all return BASIC `I/O Error`. That's the
NovaHost FileIoController layer. No FPGA changes in this area, so
probably a NovaHost firmware regression or a path/permissions issue on
the ESP32 side. Sanity-check the NovaHost SD or flash-file path config.

## Two minor followups

- `math.6502`: `array-operations` single test failure. Unknown cause;
  may be a regression from Phase 2 work or a pre-existing flake.
- Font slot 3 is currently zero-padded (`e6502.FPGA/rom/fonts.hex` has
  cp437+upper+lower+zeros). User tested `FONT 3` and saw empty chars.
  By design — Avalonia is 0-indexed too — but maybe friendlier to
  either (a) return cp437 on out-of-range slot, or (b) clamp in
  `basic.asm:LAB_FONT`. User preference unknown.

## What I wouldn't touch

- `sid_curve_reader.sv` wiring — the FPGA side was sim-tested and
  looks right. The failure is the ESP32 half of the loader, not this.
- The accumulator refactor in `blitter.sv` — the logic is equivalent
  to the prior multiplier form and sim confirms. Either there's a
  subtle synth-only issue or the DMA failure has a different root
  cause.

## Working-state preservation

Bitstream **that flashes cleanly and runs BASIC** is:
`e6502.FPGA/fpga/build/e6502.phase2_timing_fix.bit`.

If further changes break hardware, `openFPGALoader --board ulx3s
e6502.phase2_timing_fix.bit` then (new TCP connection to
`novahost.local:6503`) `{"command":"reload_rom"}` then `{"command":
"cold_start"}` restores the BASIC prompt.

NovaHost JSON field is `"command"` (not `"cmd"`). POKE/PEEK use
`"address"` and `"value"`.
