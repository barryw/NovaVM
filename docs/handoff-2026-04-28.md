## What we shipped (ALL still in working tree, NOT committed)

### 1. VGC register-slice timing fix — WORKS

Files: `e6502.FPGA/rtl/vgc.sv`, `e6502.FPGA/test/vgc_tb.svh`

- Added a 1-cycle write-side register slice (`r_cpu_addr_w`, `r_cpu_wdata_w`, `write_active`) to break the BRAM→CPU→VGC.cmd_char_addr critical path that was the original 41 ns / 24.31 MHz bottleneck.
- Duplicated decoders into `_w` variants so the write block uses registered signals.
- Bus-fabric timing: 24.31 MHz → 29.72 MHz worst-case PASS (8-seed sweep). Math copro headroom secured.
- `vgc_tb.svh::bus_write` extended with 2 extra `@(posedge clk)` for the new write latency.

### 2. SysReset (CMD `$1F`) refactor — IN CODE, BEHAVIOR INCOMPLETE

`vgc.sv`'s SysReset case now mirrors the `if (rst)` init list (cursor/colors/sprite state/copper/regs[]/etc.) and re-arms `boot_clear_armed`. Plus the framebuffer clear.

### 3. Auto-clear FSM — DISABLED (commented out) in vgc.sv

The auto-clear was supposed to dispatch TXTCLS+GCLS once per cold_start to wipe the framebuffers. It works in principle but races BASIC's cold-start banner — TXTCLS overwrites BASIC's "Ready" prompt cell-by-cell. Correct fix requires holding CPU in reset until clear completes (multi-module change). Currently commented out; framebuffer carryover between cold_starts is an open bug.

### 4. GSR primitive instantiation — IN CODE, FRESH SYNTH NEEDED IF YOU WANT TO RE-TRY

`fpga_top.sv` has `SGSR GSR_INST (.GSR(~rst), .CLK(clk_pixel))`. **This is the broken state.** SGSR resets every FF in the design globally including debug_bridge (UART clock domain) — async CDC violation wedges ROM-load. **Revert to no GSR primitive OR change to `GSR GSR_INST(.GSR(1'b1))` (POR-only) before next synth.**

## The open bug (next session's main work)

**Symptom:** HDMI rendering shifts left by 1 character (the leftmost column scrolls off-screen) after some sequence of cold_starts. Intermittent — depends on PnR placement luck. The screen reads "eady" instead of "Ready", "0 MODE 3" instead of "10 MODE 3".

**What we know:**
- The shift is RUNTIME, not POR-fixed. The board boots clean, then shifts after the test runner triggers cold_start a few times.
- Verilator simulation passes 100% — RTL logic is correct end-to-end. Bug is HW-only.
- 7831 of 7872 FFs in the synthesized design have `GSR("DISABLED")` because they have explicit `if (rst)` clauses — yosys's ECP5 GSR pass leaves those alone.
- Adding `GSR GSR_INST(.GSR(1'b1))` (POR-only) made 41 critical FFs (PLL_LOCK sync chain at fpga_top.sv:98, rst_cnt at fpga_top.sv:417) deterministic at config time. Not enough — shift recurred during suite.
- Adding global GSR/SGSR for runtime reset broke debug_bridge (multi-clock-domain CDC violation).

**Hypothesis (not verified):** the shift is caused by FFs in the visual pipeline coming up at silicon-default (random) state on the FIRST cold_start, then drifting. The 7831 GSR=DISABLED FFs are the suspects. Yosys leaves them at silicon-default until rst fires, but rst is ALSO an FF chain, so during the rst-tree's fanout window some FFs see clk edges before rst arrives.

**Concrete next steps:**
1. **Study working ECP5 references** — emard's ulx3s-misc, BrunoLevy/learn-fpga FemtoRV, lawrie/ulx3s_sms. They use the same vga2dvid + ESP32-debug architecture. Copy their reset pattern verbatim.
2. **Try forcing GSR=ENABLED on all FFs** via post-techmap yosys command. Earlier attempts:
   - `chparam -set GSR \"ENABLED\" t:TRELLIS_FF` — ran without error but didn't change values.
   - Need to understand why. Possible: chparam can't change cell parameters this way, only module parameters. Alternative: `setattr` with `init` attribute.
3. **Compare git diff with the state of `e6502.dpram_dout_init_seed3_25.72MHz_2026_04_26.bit`** which was 5/5 clean prior session. There may be a structural reset-tree difference my register-slice change introduced that we haven't identified.
4. **Consider an architectural pivot:** remove `if (rst)` clauses from EVERY FF, use declaration init only, accept that FFs come up at config time and stay there. That gives every FF a `GSR("ENABLED")` slot in the bitstream. Cost: no runtime soft-reset. Benefit: total POR determinism. Estimated: large refactor, 4-8 hours.

## NovaHost gotchas

- Port 6503 (TCP JSON): `reload_rom` (12s wait), `cold_start` (5s wait), `read_screen`. NOT `get_screen`.
- Port 23 (TCP plaintext): NovaHost log viewer. Useful when 6503 wedges to see what NovaHost saw last.
- After integration test sweeps that fail (TCP wedge), HW must be **physically power-cycled** to recover. The wedge is a known NovaHost firmware bug.
- HTTP `/health` returns `{"ok":true}` when ESP32 is responsive — quickest check before any 6503 commands.
