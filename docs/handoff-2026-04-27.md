## The structural reality

Critical path (per nextpnr `--report`):
```
BRAM clk-to-q (5.83 ns FIXED on DP16KD) →
  BRAM cascade output mux (~5 LUT levels for 64KB / 32 blocks) →
  cpu_din 3-way grouped mux (~3 LUT levels) →
  cpu_din register
```

Total logic: ~13 ns. Routing in 25-30 ns. Total: 38-40 ns — right on the 40ns/25MHz edge.

**The user vetoed:** main_ram size reduction (would shrink cascade), CPU speed reduction.

**Remaining real options for "correct" investigation** (priority order):

### 1. Verify BRAM `OUTREG` usage in post-synth
Check `e6502.FPGA/fpga/build/post_synth.v` for `DP16KD` instances. Confirm:
- `REGMODE_A` / `REGMODE_B` is set to `OUTREG` (not `NOREG`) when our `dout_a <=` / `dout_b <=` patterns are used in dpram.sv
- If yosys generated NOREG + external FF instead of using BRAM internal OUTREG, we're double-paying clk-to-q delay

If external FF is used, the path is `BRAM async dout → external FF` (BRAM async read is ~10 ns, then FF clk-to-q on external FF is ~5 ns = ~15 ns critical path **just for the read register**). vs. `BRAM internal OUTREG (5.83 ns total)`. This is a 2× difference in worst case.

```bash
grep -A 5 "DP16KD" e6502.FPGA/fpga/build/post_synth.v | grep -iE "REGMODE|CSDECODE|DATA_WIDTH" | head
```

### 2. Audit cpu_din mux input registration
Every signal feeding the cpu_din mux MUST be pre-registered (the `r_*` pattern in top.sv). If any is unregistered, it adds LUT levels to the critical path.

Currently the mux structure (top.sv:444-490):
- `xmc_group_data` ← combinational from `r_xmc_win_sel`, `r_xmc_win_enabled`, `xram_a_dout`, `r_xmc_reg_sel`, `r_xmc_reg_data`, `r_bm_cpu_rdata`. Audit: is `xram_a_dout` registered or combinational from xram_sdram?
- `io_group_data` ← `r_fio_reg_sel`, `r_fio_cpu_rdata`, `r_vgc_cpu_rdata`. All registered ✓
- `mem_group_data` ← `r_cpu_in_rom`, `r_ext_rom_active`, `erom_b_dout`, `brom_b_dout`, `ram_a_dout`. **The BRAM doutS are clocked outputs of the BRAM (via dpram), not external `r_*` regs — these are the actual critical path source.**

### 3. Verify `cpu_in_rom` comparator depth
`wire cpu_in_rom = (cpu_addr >= ROM_BASE);` should compile to bit `cpu_addr[14] | cpu_addr[15]` ≈ 1 LUT. Confirm in post_synth.v that it isn't expanded into a wide comparator chain.

### 4. Audit `held_cpu_addr` / `mem_addr` mux
`mem_addr = cpu_ce ? cpu_addr : held_cpu_addr` is on the BRAM **address** side. If this mux feeds main_ram.addr_a directly, every clock edge captures a new addr through a 2:1 mux per address bit. Verify this isn't on the critical path.

### 5. (LAST RESORT, only if 1-4 reveal nothing) RTL-level mux restructure
- Convert priority-chain sub-muxes (e.g., `xmc_group_data` 4-way priority) to one-hot OR-tree.
- Replace each peripheral's output with `(sel ? data : 8'h00)` and OR all together.
- Same correctness, log2(N) depth.
