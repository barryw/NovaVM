## Two blockers, both small

### Blocker 1 — physical: SD card not formatted as FAT32

128GB card ships as exFAT by default. ESP32 `SD_MMC` only speaks FAT32. The `format_if_mount_failed` flag is in firmware but never gets to format because `cardType` returned `NONE` last test (likely the card wasn't fully clicked in at that moment, or boot sampled GPIO12 strap wrong).

**Fix:** pop the card into a USB SD reader on the Mac:
```bash
diskutil list                                    # find /dev/diskN
diskutil unmountDisk /dev/diskN
sudo newfs_msdos -F 32 -v NOVA /dev/diskNs1
diskutil eject /dev/diskN
```
Reseat in ULX3S until it clicks. Power-cycle the board (full unplug, not OTA reset).

**Verify:**
```bash
curl http://192.168.1.65/sd-status
# expect: {"cardType":"SDHC","cardTypeId":3,"cardSize":~128000000000,...}
```

### Blocker 2 — technical: BASIC ROM is at capacity

Last night I tried to add a `WAITVBL` token (needed for game frame-sync). Every existing token slot is used; adding any new one overflows the CODE segment by ~21 bytes.

**Fix path (recommended):** move 5-10 less-used XTK_* tokens to `extension_rom` (16KB second bank, currently underused). Candidates for relocation:
- `XTK_NCC` (NCC editor — rarely used in games)
- `XTK_COPPER` (copper list — niche)
- The full `XTK_N*` networking family (not arcade-relevant)
- `XTK_MIDPLAY` / `XTK_MIDSTOP` (MIDI playback — heavy, soundfont-dependent)
- `XTK_SIDPLAY` / `XTK_SIDSTOP` (.sid file playback)

Likely frees 200+ bytes — room for a generation of new tokens.

Other options if relocation is too invasive:
- Audit `basic.asm` for dead EhBASIC heritage code (probably 100+ bytes recoverable)
- Expand ROM region in `ehbasic.cfg` (requires top.sv RTL change to map more)
